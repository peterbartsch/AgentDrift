## Findings

### Critical

#### CRT-1 — DESIGN.md self-contradiction on canonical brand gold

- **Severity:** Critical
- **Location:** `DESIGN.md:35` ↔ `DESIGN.md:440`
- **Token:** brand secondary / gold
- **Values observed:**
  - DESIGN.md Section 2, line 35: `#E8AF00` (canonical, dated 2026-04-20)
  - DESIGN.md Section 9 ("Agent Prompt Guide"), line 440: `Gold (#F0B800)`
- **Why it matters:** Section 9 is the page agents read to know what color to use. It currently teaches `#F0B800`, which production no longer uses. Any agent generating new UI from this guide will hardcode the wrong gold.
- **Recommended fix:** Update DESIGN.md:440 to read `Gold (#E8AF00)` to match Section 2.
- **Loop step:** 6 (Document — the rules layer is out of sync with itself)

#### CRT-2 — "Premium gold" #D4A000 has no entry in tokens.json

- **Severity:** Critical
- **Location:** `DESIGN.md:39`, `DESIGN.md:159` ↔ `lib/styles/tokens.json` (no entry)
- **Token:** premium gold (Founders Edition tier)
- **Values observed:**
  - DESIGN.md Section 2, line 39: "Premium gold" `#D4A000` documented as a real brand color for premium tier
  - DESIGN.md Component Stylings, line 159: `.hero-cta-primary` references `#D4A000` for premium variant
  - tokens.json: no `gold.600`, no `premiumGold`, no semantic `brand.premiumSecondary` entry
- **Why it matters:** A documented brand color used by a real component lives only in DESIGN.md and inline CSS. Tokens Studio cannot import it into Figma; Figma users have no Variable to bind to. The premium tier is invisible in the bridge layer.
- **Recommended fix:** Add `primitive.color.gold.600 = #D4A000` and `semantic.color.brand.secondaryPremium → {primitive.color.gold.600}` to tokens.json. Add `--color-secondary-premium: #D4A000` to main.css `:root`. Re-run skill to confirm.
- **Loop step:** 3 (Tokenize)

### High

#### HIGH-1 — Auxosphere brand color drift

- **Severity:** High
- **Location:** `DESIGN.md:79` ↔ `lib/styles/tokens.json:192`
- **Token:** sphere.auxosphere
- **Values observed:**
  - DESIGN.md Section 2, line 79: `Auxosphere | Gray | #909090`
  - tokens.json line 192: `"auxosphere": { "$value": "{primitive.color.gray.500}" }` which resolves to `#6c757d`
- **Why it matters:** The Auxosphere product page on production may render a gray that the spec does not endorse. Six sphere brands are documented; one of them disagrees with itself across the rules layer and the bridge.
- **Recommended fix:** Decide canonical value. Either:
  - (a) Add `primitive.color.gray.400 = #909090` and rebind `sphere.auxosphere → {primitive.color.gray.400}`, OR
  - (b) Update DESIGN.md:79 to `#6c757d` and align with the existing primitive.
  - Recommendation: (a) — `#909090` is a distinct brand-accent gray, not the body-text neutral; merging them flattens the brand.
- **Loop step:** 3 (Tokenize)

### Medium

#### MED-1 — design-system.html missing accessible-gold-on-white token

- **Severity:** Medium
- **Location:** `lib/styles/main.css :root` ↔ `design-system.html :root`
- **Token:** `--color-secondary-text` (gold.900, `#946E00`, WCAG AA on white)
- **Values observed:**
  - main.css: `--color-secondary-text: #946E00;` declared
  - design-system.html: not declared in inline `:root`
- **Why it matters:** The accessibility-correct gold-on-white text color is unavailable to anything composed on the design-system.html demo page, so any new component documented there cannot use it.
- **Recommended fix:** Add `--color-secondary-text: #946E00;` to design-system.html `:root`.
- **Loop step:** 6 (Document)

#### MED-2 — design-system.html missing breakpoint tokens

- **Severity:** Medium
- **Location:** `lib/styles/main.css :root` ↔ `design-system.html :root`
- **Tokens:** `--breakpoint-xs` through `--breakpoint-2xl` (six tokens, 480 / 640 / 768 / 1024 / 1440 / 1920px)
- **Values observed:** Declared in main.css, absent from design-system.html
- **Why it matters:** Component examples on design-system.html cannot reference responsive breakpoints by token. Any responsive demo added there will hardcode pixel values, teaching the wrong pattern.
- **Recommended fix:** Copy the six `--breakpoint-*` declarations from main.css `:root` into design-system.html `:root`.
- **Loop step:** 6 (Document)

#### MED-3 — design-system.html z-index scale truncated

- **Severity:** Medium
- **Location:** `lib/styles/main.css :root` ↔ `design-system.html :root`
- **Tokens:** `--z-popover` (1060), `--z-tooltip` (1070), `--z-skip-link` (1080)
- **Values observed:** Declared in main.css, absent from design-system.html (which stops at `--z-modal: 1050`)
- **Why it matters:** Three z-index layers exist in production but are invisible to anyone consulting the demo. New popovers/tooltips/skip-links documented here will collide with modals.
- **Recommended fix:** Add the three missing z-index declarations to design-system.html `:root`.
- **Loop step:** 6 (Document)

### Low

#### LOW-1 — `--color-accent-dark: #1e7e34` aliasing in main.css

- **Severity:** Low
- **Location:** `lib/styles/main.css :root` (`--color-accent-dark: #1e7e34`)
- **Token:** accent dark (mapped to primitive.color.green.600)
- **Values observed:**
  - main.css: `#1e7e34` declared as `--color-accent-dark`
  - tokens.json: `primitive.color.green.600 = #1e7e34` exists, but `semantic.color.brand.accentActive → {primitive.color.green.600}` is the documented binding
- **Why it matters:** The same hex `#1e7e34` was historically the value of an *incorrect* `--color-primary-light` (per `tokens.json $metadata.cssSyncRequired` history). The value is correct here, but the aliasing of green to a "primary" name family in the past was the bug. No production issue today; flagging for vigilance.
- **Recommended fix:** None required. Document in `tokens.json $metadata.brandDecisions` that `#1e7e34` belongs only to `green.600`.
- **Loop step:** 3 (Tokenize)
